## 一、项目概述

土豪龙虾酒吧是一个基于 Web 的模拟经营类应用,用户扮演酒馆老板接待VIP客户、管理订单和经营酒馆。应用采用 React + Zustand 技术栈,提供实时互动、数据分析和身份升级等核心功能。

### 核心功能模块
- **点单系统**: 酒水、海鲜、请全场三大消费场景
- **订单管理**: 实时订单处理、状态跟踪、历史记录
- **身份体系**: 普通会员、钻石会员、至尊会员三级身份
- **数据运营**: KPI仪表盘、排行榜、消费分析
- **用户互动**: 签到奖励、社交裂变、动态播报

## 二、Bug 修复需求

### Bug-001: 注册名重复问题
**优先级**: 高 | **状态**: 已修复 | **文件**: `barStore.js`、`WelcomePage.jsx`

**问题描述**:
用户注册时可以使用与现有用户(包括mock用户)相同的昵称,导致数据混乱和显示冲突。

**复现步骤**:
1. 打开欢迎页,点击"使用新账号入场"
2. 在昵称输入框输入与现有用户相同的名字(如"龙虾界大佬")
3. 点击"入场"按钮,系统成功创建账号

**解决方案**:
1. 在 `barStore` 中新增 `checkNameAvailable(name)` 工具函数,检查昵称是否可用
2. 在 `WelcomePage` 中添加实时昵称验证和错误提示
3. 当昵称重复时禁用入场按钮并显示红色错误提示

**验收标准**:
- 用户无法使用与mock用户相同的昵称注册
- 实时输入时检查并显示错误提示
- 输入框边框在有错误时变红
- 入场按钮在昵称重复时禁用

### Bug-010: 用户重复注册数据被覆盖 + 用户名唯一性
**优先级**: 高 | **状态**: 已修复 | **文件**: `barStore.js`、`WelcomePage.jsx`

**问题描述**:
1. 已注册的用户A消费后,再次注册相同名字A,所有数据(算力、订单、消费)被重置
2. 用户无法直接登录已有账号,只能通过"使用新账号入场"重新创建
3. 用户名无法保证唯一性,历史注册过的用户名可以被再次注册

**复现步骤**:
1. 注册用户A,消费后退出
2. 再次点击"使用新账号入场",输入用户A
3. 进入后发现算力、订单等数据全部重置
4. 没有直接登录已注册用户的入口
5. 用户B可以注册与用户A相同的名字

**解决方案**:
1. **修复 `checkNameAvailable` 函数**: 删除重复逻辑,增加历史用户名检查
2. **新增 `registeredUserNames` 状态**: 维护所有注册过的用户名列表
3. **新增 `isUserRegistered` 函数**: 检查用户名是否为当前已注册用户
4. **修改 WelcomePage 登录逻辑**:
   - 检测输入用户名是否为已注册用户
   - 如果是,直接登录(保持原有数据)
   - 如果否,检查是否可用后创建新用户
5. **增强数据持久化**: 添加 `registeredUserNames` 字段持久化
6. **修改 `createUser` 函数**: 创建用户时将用户名添加到注册列表
7. **修改 `logout` 函数**: 退出时保留注册用户名列表

**验收标准**:
- 已注册用户重新登录时保持原算力、订单、消费记录
- 用户名保证全局唯一性(包括历史注册用户)
- 购物车、消息等状态在刷新后保持
- 新用户注册不受影响
- 用户可以无缝登录已有账号
- 用户退出后用户名仍被占用,不可再次注册

### Bug-011: 在线人数显示不一致
**优先级**: 中 | **状态**: 已修复 | **文件**: `barStore.js`、`adminStore.js`

**问题描述**:
前端首页显示在线人数为28人,后台首页显示7人,数据不一致且都是硬编码值,不能反映真实在线状态。

**复现步骤**:
1. 前端首页显示"当前酒馆已有 28 位土豪在场"
2. 登录后台查看KPI卡片显示"当前在线 7 位土豪"
3. 两者数据不一致且与实际用户状态不匹配

**解决方案**:
1. **前端修复**: 将 `barStore.onlineCount` 从硬编码28改为动态计算
   - 初始值为8个bot用户
   - 登录时: `onlineCount = 8 + 1 = 9`
   - 退出时: `onlineCount = 8`
2. **后台修复**: 
   - 修正 `MOCK_USERS` 所有用户状态为 `'online'`
   - 在 `syncFromBar` 中动态计算 `stats.onlineUsers`
   - 根据实际用户列表中 `status === 'online'` 的数量更新

**验收标准**:
- 前端首页显示正确的在线人数(8+登录用户数)
- 后台首页KPI卡片显示与前端一致的在线人数
- 用户登录/退出时在线人数实时更新
- 所有mock用户默认在线

### Bug-002: 新用户登录订单残留问题
**优先级**: 高 | **状态**: 已修复 | **文件**: `adminStore.js`

**问题描述**:
当新用户(已登录但未下单)进入系统时,会残留旧用户的订单数据,导致数据混乱。

**复现步骤**:
1. 用户A下单后退出登录
2. 用户B登录系统但尚未下单
3. 观察订单列表,发现显示用户A的订单

**解决方案**:
在 `syncFromBar` 函数中补充 `else if (user && barOrders.length === 0)` 分支,将订单列表重置为 MOCK_ORDERS。

**验收标准**:
- 新用户登录后订单列表为空或显示默认测试数据
- 已有用户的订单数据不受影响

### Bug-012: 订单ID重复问题
**优先级**: 高 | **状态**: 已修复 | **文件**: `barStore.js`

**问题描述**:
使用 `Date.now()` 作为订单ID,当用户快速操作时可能产生重复key导致系统异常。

**复现步骤**:
1. 连续快速创建多个订单(短时间内)
2. 观察控制台报错或订单数据异常

**解决方案**:
将所有订单ID生成从 `Date.now()` 改为 `crypto.randomUUID()`,确保唯一性。

**验收标准**:
- 批量创建订单无重复key错误
- 订单ID格式为标准UUID v4

### Bug-013: Zustand Store排序污染
**优先级**: 中 | **状态**: 已修复 | **文件**: `Dashboard.jsx`

**问题描述**:
直接对 Zustand store 中的 `users` 数组进行 `sort()` 原地排序,导致状态被污染。

**复现步骤**:
1. 打开 Dashboard 页面
2. 观察用户列表排序变化
3. 多次刷新页面,排序结果不一致

**解决方案**:
将 `users.sort()` 改为 `[...users].sort()`,创建副本后再排序。

**验收标准**:
- 用户排序功能正常
- 原始 store 数据不被修改
- 刷新页面排序结果一致

### 优化-023: 低算力自动预警
**优先级**: 中 | **状态**: 已实现 | **文件**: `MainPage.jsx`

**业务目标**:
提前预警用户算力不足,促进身份升级转化。

**功能需求**:
1. 实时监控用户算力变化
2. 算力首次降至100以下时自动弹出Toast警告
3. 算力回升到100以上后重置预警状态
4. 下次降至100以下时再次触发预警

**技术实现**:
- 使用 `useRef` 存储 `lowPowerAlerted` 标志
- 使用 `useEffect` 监听 `power` 状态变化
- 判断逻辑: `power < 100 && !lowPowerAlerted.current`
- 触发预警后设置 `lowPowerAlerted.current = true`
- 算力回升时重置标志

**验收标准**:
- 算力降至99时立即触发预警
- 预警只触发一次,不重复弹窗
- 算力回升后下次降阈值仍可预警
- Toast文案清晰准确

**数据指标**:
- 低算力预警触发率
- 预警后升级转化率
